# Architecture Documentation

## Architecture Evolution

This document tracks the architectural decisions made for the Order and Payment Microservices platform, specifically during the migration from REST to gRPC in the second assignment.

## Phase 2: gRPC Migration & Contract-First Design

### 1. Contract-First Protocol Buffers
- **Decision:** Moved all inter-service contracts to a unified external repository using Protocol Buffers (`.proto`).
- **Rationale:** Strict typing, cross-language support, schema validation before deployment.
- **Workflow:** Set up two GitHub repositories:
  - **Repo A:** Holds `.proto` files (`ap2-contracts`).
  - **Repo B (`youruser/repo-b`):** Receives code natively generated by GitHub Actions. Services import this Go module directly.

### 2. Internal vs. External APIs (Order Service)
- **Decision:** Retained Gin REST API (`:8080`) for the Order Service client-facing endpoints `POST /orders`, etc.
- **Decision:** Integrated gRPC as the sole protocol for internal synchronous communication (e.g. Order Service → Payment Service ProcessPayment).
- **Decision:** Integrated a gRPC Server Streaming RPC on the Order Service (`:50051`) for `SubscribeToOrderUpdates`, replacing polling for real-time order tracking.

### 3. Server-Side Streaming (Order Tracking)
- **Decision:** Implemented a Server-Side Streaming RPC (`SubscribeToOrderUpdates`) in the Order Service.
- **Rationale:** Demonstrates true, real-time push-based notifications without client polling.
- **Mechanism:** The server queries the database when a stream is open, pushing new messages to the client over an open TCP connection when `order.Status` is changed.

### 4. Logging Interceptor (Bonus)
- **Decision:** Added a Unary Interceptor on the Payment Service gRPC Server.
- **Rationale:** Automatically logs method names and request durations for observability across the microservice boundary without polluting use-case layers.

## Clean Architecture Structure

### Layer Separation
1. **Domain**: Defines business constraints and behavior. Unaware of the outside world.
2. **Use Case**: Coordinates domain logic, delegates data-access to the repository interface.
3. **Repository**: Deals with SQL implementation or exterior network clients (HTTP or gRPC).
4. **Transport**: Deals with framework specific implementations (Gin/REST or `grpc.Server`).

## Dependency Flow (Inward - Clean Architecture)

```
                    DOMAIN
                      ▲
                      │
        ┌─────────────┼─────────────┐
        │             │             │
        │             │             │
    USE CASE    USE CASE    USE CASE
        │             │             │
        └─────────────┼─────────────┘
                      │
                      ▲
        ┌─────────────┼─────────────┐
        │             │             │
    REPOSITORY   REPOSITORY   REPOSITORY
    (DB)         (Http)       (Cache)
        │             │             │
        └─────────────┼─────────────┘
                      │
                      ▲
                 DELIVERY/HTTP
```

**Key Rule**: Outer layers can depend on inner layers, but inner layers
can NEVER depend on outer layers. All dependencies point inward.
